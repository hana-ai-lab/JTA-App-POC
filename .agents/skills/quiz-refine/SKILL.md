# Quiz Refine Skill

既存クイズを「退場しないトレーダーを育てる」という教育目的に沿って改善するスキル。
定義型・計算型から、**シナリオ型・結果型**へのリライトを軸に、章テーマの定着を促す。

## ファイル構成

```
docs/quizzes/{course}/week-{NN}.md   # 改善対象のクイズMD
server/seeds/001_quiz_data.sql       # SQL seedファイル（ローカル）
docs/knowledge/                      # ナレッジファイル（講義内容の参照元）
docs/lectures/                       # 講義資料（存在する場合）
```

## ステップ

### 1. 対象の確認

ユーザーが指定したコース・章・weekを確認する。未指定の場合は AskUserQuestion で聞く。

- 例: `kohaku week-01`（入門コース 第1週）
- 対象MD: `docs/quizzes/kohaku/week-01.md`

### 2. 章テーマの特定

講義資料・ナレッジファイルを読み、**この章で学習者に体得させたい核心的な考え方**を1〜2文で定義する。

参照する順序:
1. `docs/knowledge/` 配下の該当コース・週のファイル
2. `docs/lectures/` 配下の資料（存在する場合）
3. クイズMD内のフロントマターの `title` やコメント

**章テーマ例（参考）:**

| コース | Week | 章テーマ |
|--------|------|----------|
| kohaku | 01   | 「退場する理由を理解し、自分がそうならないようにする」— バイアスによる負け方を体感的に理解する |
| kohana | 01   | 「待つことがトレーダーの最大の武器」— 個人投資家のアドバンテージを腹落ちさせる |

### 3. 既存クイズの診断

対象MDを読み込み、各問題を以下の3タイプに分類する:

| タイプ | 定義 | 判断 |
|--------|------|------|
| 応用型（シナリオ） | 具体的な状況・行動・結果を問う | ✅ 維持または改善 |
| 定義型 | 「〜とは何ですか？」「〜の特徴は？」 | ⚠️ シナリオ型に書き換え |
| 計算型（重複） | 同じ概念を数字を変えて繰り返す | ⚠️ 概念問題に置き換え |

**診断観点:**
- 章テーマと無関係な問題はないか
- 同じ知識を問う問題が重複していないか
- 正解の文言が循環的でないか（例：「待てるから」→ 理由になっていない）
- **「用語・分類を当てる問題」になっていないか**（下記参照）

#### ⚠️ 「用語・分類を当てる問題」は作らない

バイアス名や心理的罠の名前を「当てる」問題は、知識の暗記にしかならず、実際のトレードで役に立たない。

| NG問い | 理由 |
|--------|------|
| 「この行動は何というバイアスですか？」 | 名前を知っても行動は変わらない |
| 「どの心理的罠に分類されますか？」 | 分類できても損切りできるようにはならない |
| 選択肢が「確証バイアス — 定義文」の形式 | バイアス名の暗記になる |
| 「退場のパターンとして正しくないものは？」 | パターン名の暗記になる |

**問いの軸は常に「負ける理由・メカニズム・取るべき行動」に置く:**

| ✅ 良い問いの軸 | 例 |
|----------------|-----|
| なぜ負けたのか | 「この行動の根本にある仕組みはどれですか？」 |
| どうなるのか | 「この状態を続けるとどのような結果に陥りますか？」 |
| 何をすべきか | 「この状況で取るべき行動として正しいものはどれですか？」 |
| 何が危険なのか | 「この思考の何が危険ですか？」 |

#### ⚠️ 退場パターンを題材にする場合の特別基準

「退場パターンの名前を知る」ことが目的ではなく、「自分がそのパターンに向かう行動をしていないか気づける」ことが目的。

各パターンを問う場合、**「危険な行動」→「その結果」→「だからこうする」** の一本線を問いの中に持たせる:

| 退場パターン | 問うべき危険な行動 | 答えにつながる行動変容 |
|-------------|-------------------|----------------------|
| 一発退場型 | フルレバ・大きすぎるポジション | 1回のリスクを資金の数%以内に制限する |
| ジリ貧型 | 振り返りなし・損益比の確認なし | 毎週勝ち負けの平均損益を記録・比較する |
| 休眠型 | 精神的に耐えられないロットでのギャンブル | 次のトレードを冷静に続けられるロットサイズにする |

**退場パターン問題のNG・OK例:**

| NG（パターン名当て） | OK（行動シナリオ） |
|---------------------|-------------------|
| 「退場のパターンとして正しくないものは？」 | 「全力でエントリーして大損後にチャートが開けなくなった。根本原因は？」 |
| 「最も多い退場パターンはどれですか？」 | 「勝率65%なのに口座が減り続けている。何が起きているか？」 |

### 4. 問題のリライト原則

#### 行動変容直結の原則（最重要）

クイズを読んだ直後に「だから自分はこうすべき/してはいけない」が腹落ちする形にする。

**正解の構造:**
- `何がNGか（メカニズム）→ その結果（損失・失敗）` の一本線
- または `何がNGか → だから正しい行動は〇〇` の形
- どちらか一方でも「行動への接続」が見えていること

**解説の末尾チェック:**

| NG（観察・分類で終わる） | OK（行動・結果で締める） |
|--------------------------|--------------------------|
| 「〜という認知バイアスが原因です」 | 「〜に縛られると損切りタイミングを失い、損失が膨らむ」 |
| 「〜という心理が働いています」 | 「〜と感じたとき、それがサインで即座に損切りルールを確認する」 |
| 「〜のが典型パターンです」 | 「〜するだけで〜できなくなるため、エントリー前に〇〇を決める」 |

#### 問題文の書き方
- **シナリオで始める**: 「あるトレーダーが〜した。」「〜という状況で〜した結果、」
- **結果を問う**: 「なぜ〜したか？」「どうなりますか？」「何が問題か？」
- 用語の定義を直接聞かない

#### 選択肢の書き方
- 正解は具体的な理由や仕組みを述べる（循環表現NG）
- 誤答は「ありがちな誤解」に基づく自然なひっかけにする
- 全選択肢の文体・長さを揃える

#### 解説の書き方
- 正解の**なぜ**を一言で説明する
- 講義の核心ポイントと接続する
- **末尾は必ず「行動・結果」で締める**（観察や分類で終わらせない）

#### 文字数ガイドライン

| 要素 | 目安 | 上限 |
|------|------|------|
| 問題文 | 60〜100文字 | 120文字 |
| 選択肢 | 20〜40文字 | 50文字 |
| 解説 | 80〜120文字 | 150文字 |

#### Before / After 例

**悪い例（定義型）:**
```
問題: 損失回避バイアスとは何ですか？
正解: 損失の痛みを利益の喜びより強く感じる心理傾向
```

**良い例（シナリオ型）:**
```
問題: あるトレーダーが含み損−5万円のポジションを持ち続けている。「もう少し待てば戻るはず」と感じている。この行動の根本にある心理は？
正解: 損失が確定する痛みが、含み損を保有し続けるコストより強く感じられているから
```

**悪い例（循環解答）:**
```
問題: 個人投資家が機関投資家より有利な点は？
正解: 待てるから
```

**良い例（理由が明示）:**
```
問題: 個人投資家が機関投資家より有利な点は？
正解: 四半期報告などのパフォーマンス義務がなく、条件が揃った最良の場面だけを選んでエントリーできるから
```

### 5. MDファイルの更新

`docs/quizzes/{course}/week-{NN}.md` を Edit ツールで更新する。

変更が多い場合は Node.js スクリプトではなく、Edit ツールで1問ずつ確実に更新する。

### 6. SQLファイルの更新

`server/seeds/001_quiz_data.sql` の対応する INSERT 文を Edit ツールで更新する。

注意事項:
- シングルクォートは `''` でエスケープする
- `correct_answer` は 1〜4 の整数（A=1, B=2, C=3, D=4）
- 問題数が変わらないことを確認（各章20問）

### 7. VPS DB の直接更新（ユーザー進捗を保持するため）

`npm run seed` はユーザー進捗データを**全消去**するため使用しない。
代わりに SSH 経由で直接 UPDATE する。

**VPS 接続情報:**
```
host:     85.131.249.99
user:     root
password: hana87tradeer-!
DB container: jta-db
DB user:  postgres
DB name:  jta
```

**重要: ファイル作成と expect 実行は1つの Bash コマンドにまとめる**

`/tmp` ファイルは Bash コマンド間で消えることがある。
SQL ファイル作成・exp ファイル作成・実行を `&&` で1コマンドに連結する。

```bash
cat > /tmp/quiz_update.sql << 'SQLEOF'
UPDATE quizzes SET
  question = '新しい問題文',
  choice_1 = '選択肢1',
  choice_2 = '選択肢2',
  choice_3 = '選択肢3',
  choice_4 = '選択肢4',
  correct_answer = 1,
  explanation = '解説文'
WHERE chapter_id = 1 AND sort_order = 3;
SQLEOF
cat > /tmp/run_sql.exp << 'EOF'
#!/usr/bin/expect -f
set password "hana87tradeer-!"
spawn bash -c "cat /tmp/quiz_update.sql | ssh -o StrictHostKeyChecking=no root@85.131.249.99 'docker exec -i jta-db psql -U postgres -d jta'"
expect "password:"
send "$password\r"
expect eof
EOF
chmod +x /tmp/run_sql.exp && expect /tmp/run_sql.exp
```

成功すると各 UPDATE ごとに `UPDATE 1` が出力される。

**複数問の一括更新時は、全 UPDATE を1つの SQL ファイルにまとめて一度に実行する。**

### 8. 反映確認

VPS DB の更新後、内容を確認する（DB名は `jta`）:

```bash
cat > /tmp/verify.exp << 'EOF'
#!/usr/bin/expect -f
set password "hana87tradeer-!"
spawn ssh -o StrictHostKeyChecking=no root@85.131.249.99 "docker exec jta-db psql -U postgres -d jta -c \"SELECT sort_order, LEFT(question, 40), correct_answer FROM quizzes WHERE chapter_id=CHAPTER_ID ORDER BY sort_order;\""
expect "password:"
send "$password\r"
expect eof
EOF
chmod +x /tmp/verify.exp && expect /tmp/verify.exp
```

（`CHAPTER_ID` を実際の値に置換して実行）

### 9. 報告

ユーザーへの最終報告に含める内容:
- 変更した問題数（「20問中X問を書き換えました」）
- 診断結果（定義型X問→シナリオ型に変換、重複計算X問→概念問題に変換）
- 章テーマとの整合性の評価
- VPS DB 更新の成否

## 注意事項

- 問題文・解説に**固有名詞（講師名など）を使用しない**。「あるトレーダー」などの匿名表現にする
- 講義資料に記載されていない内容を**創作しない**（ファクトは必ず出典から）
- 正解の文言が「〜だから」で終わる場合、循環していないか必ず確認する
- フル資金・特定の数字・断定的な表現は講義資料で裏付けてから使う
- 文字数が超過する場合は情報を削るのではなく、**より本質的な表現に凝縮する**
